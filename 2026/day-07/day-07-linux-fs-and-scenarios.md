# Day 07 – Linux File System Hierarchy & Scenario-Based Practice

## Part 1: Linux File System Hierarchy

### Core Directories (Must Know)

#### `/` – Root
- The starting point of the entire Linux file system. Every other directory hangs off this.
- `ls -l /` shows: bin, boot, dev, etc, home, lib, media, mnt, opt, proc, root, run, srv, sys, tmp, usr, var
- I would use this when I need to navigate to any other part of the system, or understand the full path of a file.

#### `/home` – User Home Directories
- Contains personal directories for each regular user (e.g., `/home/dinesh`). Users store their files, configs, and projects here.
- `ls -l /home` shows user directories like `dinesh/`, each owned by that user.
- I would use this when managing user files, deploying apps owned by service users, or troubleshooting permission issues.

#### `/root` – Root User's Home
- The home directory of the `root` (superuser) account. Separate from `/home` for security.
- `ls -la /root` shows: `.bashrc`, `.bash_profile`, `.ssh/`, `.ansible/`
- I would use this when working as root to store scripts or check root-owned configurations.

#### `/etc` – Configuration Files
- Contains system-wide configuration files for all applications and services. No binaries here — only config text files.
- `ls /etc` shows: `hostname`, `fstab`, `ssh/`, `nginx/`, `cron.d/`, `passwd`, `sudoers`
- I would use this when configuring services, checking hostnames, editing SSH configs, or managing users via `/etc/passwd`.

#### `/var/log` – Log Files
- Contains log files generated by the OS and applications. Critical for debugging and monitoring.
- `du -sh /var/log/* 2>/dev/null | sort -h | tail -5` shows the largest logs on this system:
  ```
  23M   /var/log/messages-20260215
  23M   /var/log/messages-20260301
  26M   /var/log/sa
  37M   /var/log/audit
  74M   /var/log/ansible_logFile.log
  ```
- I would use this when investigating service failures, security incidents, or performance issues.

#### `/tmp` – Temporary Files
- Stores temporary files that are wiped on reboot. Any process can write here (world-writable).
- `ls /tmp` shows short-lived files from running processes.
- I would use this when scripts need a scratch space, or when downloading files temporarily before processing.

---

### Additional Directories (Good to Know)

#### `/bin` – Essential Command Binaries
- Contains the core commands needed for system boot and basic operation: `ls`, `cat`, `cp`, `mv`, `rm`, `bash`.
- `ls /bin | head -10`: `[`, `2to3`, `2to3-3.6`, `a2x`, `a2x.py`, `ac`, `aclocal`, `aclocal-1.16`, `addr2line`
- I would use this to find where basic commands live, or when writing scripts that need absolute paths.

#### `/usr/bin` – User Command Binaries
- Contains most user-facing commands not needed for boot: `git`, `python3`, `curl`, `wget`, `vim`, `docker`.
- `ls /usr/bin | wc -l` typically shows 1000+ binaries.
- I would use this when a command isn't in `/bin`, or when checking which version of a tool is installed.

#### `/opt` – Optional/Third-Party Applications
- Used for software installed outside the package manager — e.g., databases, monitoring agents, vendor tools.
- `ls /opt` on this system shows: `containerd`, `okta`, `radiator`
- I would use this when installing custom software that shouldn't mix with OS-managed paths.

## Part 2: Scenario-Based Practice

### Solved Example: Check if Nginx Service is Running

**Step 1:** Check service status
```bash
systemctl status nginx
```
**Why:** Shows if it's active, failed, or stopped, and the last few log lines.

**Step 2:** List all services if not found
```bash
systemctl list-units --type=service | grep nginx
```
**Why:** Confirms whether nginx is installed at all.

**Step 3:** Check if enabled on boot
```bash
systemctl is-enabled nginx
```
**Why:** Tells me if it'll restart after a server reboot.

### Scenario 1: Service Not Starting

**Problem:** Web application service `myapp` failed to start after server reboot.

**Step 1:** Check if service is running or failed
```bash
systemctl status myapp
```
**Why:** Gives the current state and the most recent log output — usually enough to spot the issue.

**Step 2:** View full logs for the service
```bash
journalctl -u myapp -n 50
```
**Why:** Shows the last 50 log lines from the service — may reveal an error like "port already in use" or "config file not found".

**Step 3:** Check if service is enabled
```bash
systemctl is-enabled myapp
```
**Why:** If it's `disabled`, it won't auto-start on reboot — that might be the whole problem.

**Step 4:** Try starting it manually and watch for errors
```bash
systemctl start myapp && systemctl status myapp
```
**Why:** Forces a start attempt so we can capture live error output.

### Scenario 2: High CPU Usage

**Problem:** Application server is slow. SSH'd in to investigate.

**Step 1:** Check live CPU usage
```bash
top
```
**Why:** Shows all processes sorted by CPU usage in real-time. Press `P` to sort by CPU.

**Step 2:** Find top CPU consumers without interactive mode
```bash
ps aux --sort=-%cpu | head -10
```
**Why:** Non-interactive, scriptable. Shows the 10 processes using the most CPU with their PIDs.

**Step 3:** Note the PID and check what process it is
```bash
ps -p <PID> -o pid,ppid,cmd,%cpu,%mem
```
**Why:** Gives detail on that specific process — parent PID and full command line.

---

### Scenario 3: Finding Service Logs

**Problem:** Developer asks "Where are the logs for the docker service?"

**Step 1:** Check service status
```bash
systemctl status docker
```
**Why:** Often shows the last few log lines right in the status output.

**Step 2:** View full logs
```bash
journalctl -u docker -n 50
```
**Why:** journald is the log collector for all systemd services. `-n 50` limits to last 50 lines.

**Step 3:** Follow logs in real-time
```bash
journalctl -u docker -f
```
**Why:** `-f` is like `tail -f` — useful when watching a service during a restart or deploy.

---

### Scenario 4: File Permissions Issue – Permission Denied

**Problem:** Script at `/home/user/backup.sh` won't run: `Permission denied`

**Step 1:** Check current permissions
```bash
ls -l /home/user/backup.sh
```
**Look for:** `-rw-r--r--` — notice no `x` (execute) bit. That's the problem.

**Step 2:** Add execute permission
```bash
chmod +x /home/user/backup.sh
```
**Why:** Adds execute permission for the owner (and group/others by default with `+x`).

**Step 3:** Verify the fix
```bash
ls -l /home/user/backup.sh
```
**Look for:** `-rwxr-xr-x` — the `x` is now present.

**Step 4:** Run the script
```bash
./backup.sh
```

---

## Key Takeaways

1. **Know your directories** — `/etc` for configs, `/var/log` for logs, `/opt` for third-party tools. These are the first places you check in any incident.
2. **Troubleshooting flow matters** — always start with `systemctl status`, then `journalctl`, then dig deeper. Don't jump to conclusions.
3. **File permissions are foundational** — a script failing with "Permission denied" is almost always a missing execute bit. Check with `ls -l` first.
